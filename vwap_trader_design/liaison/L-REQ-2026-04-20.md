## 질의 1 — Agent E (Plan Custodian)

### 주제
백테스트 데이터 소스 및 기간의 명세 완결성

### 배경
PLAN.md 부록 L.1 에 다음과 같이 명시됨:
```
데이터 소스: CCXT (bybit) + Parquet 로컬 캐싱
기간: 2022-01-01 00:00 UTC ~ 2024-12-31 23:59 UTC
```

그러나 현재 구현체 [vwap_trader/requirements.txt](vwap_trader/requirements.txt) 는 `pybit`만 포함, CCXT 미포함. Parquet 라이브러리(pyarrow/fastparquet)도 미포함.

### 질문
1. CCXT 지정은 **강제 요구사항**인가, 아니면 "Bybit 데이터를 받을 수 있는 구현체"라는 추상 명세인가?
2. pybit 으로 동일 데이터 수집이 가능할 때도 CCXT 전환이 필요한가?
3. Parquet 필수인가, CSV/pickle 로컬 캐시로 충분한가? (단위 테스트 용이성 관점)

### 영향
- 강제 요구사항이면 → requirements.txt 수정 + BybitClient 재작성 필요 (2~3일 지연)
- 추상 명세면 → 현재 pybit 기반으로 진행 (즉시 시작 가능)

## 질의 2 — Agent D (Pragmatist)

### 주제
DRY_RUN 100건의 "건(件)" 정의

### 배경
PLAN.md Chapter 10.2 "DRY_RUN 구조" 및 Chapter 0.3 "성공 기준" 에 **"DRY_RUN 100건 누적 시점"** 이 반복 명시. 그러나 "건"의 단위가 모호:

- (a) 진입 **시도** 건 수 (신호 발생 시점 기준, 주문 거부 포함)
- (b) 체결 건 수 (주문 성공 시점)
- (c) 라운드트립 완료 건 수 (진입 + 청산 전체 완료)

### 질문
1. 위 3개 중 "건"의 정의는 무엇인가?
2. 답이 (c)인 경우, max_hold 강제 청산도 "완료 1건"으로 카운트 하는가?
3. RiskManager 에 의해 거부된 신호는 어느 쪽에도 포함되지 않는 것이 맞는가?

### 영향
- (a)/(b)/(c) 선택에 따라 DRY_RUN 종료 시점이 최대 3배 차이. Chapter 10 → Chapter 11(실전) 전환 일정 산정 직결.

## 질의 3 — Agent F (Final Authority)

### 주제
백테스트 스코어 미달 시 파라미터 튜닝 허용 범위

### 배경
PLAN.md 부록 L.4 스코어 함수 + L.3 Grid Search 범위는 확정됨. 그러나 Grid 전체에서 자동 탈락 조건(`pf<1.0 or mdd>0.20`)만 통과하는 조합이 나오지 않을 경우의 처리 방침이 명시 안 됨.

### 질문
1. Grid 전체가 자동 탈락 조건 모두 불통과 시:
   - (a) Grid 범위를 **확장**하여 재탐색 (부록 L.3 의 범위 자체를 넓힘)
   - (b) 부록 A 확정 파라미터(ATR 1.5%, EMA slope 0.3% 등) **재조정**
   - (c) 전략 자체 **폐기** (Chapter 12 실패 시나리오 #1 트리거)
   - 위 중 무엇이 기본 정책인가?
2. 만약 (b)가 허용된다면, 재조정 시 기획팀 재승인이 필요한가, Dev-PM 재량인가?
3. 상위 조합의 PF 가 1.0 초과하지만 1.2 미만처럼 "통과하지만 실전 BEP 미달"인 경우 처리?

### 영향
- 백테스트 결과 해석의 가장 중요한 분기. 답 없이 결과 나오면 해석 보류 → 전체 일정 무기한 지연.

## 질의 4 — Agent C (Volume Profile 이지원)

### 주제
`va_slope` 의 정의 및 계산 로직 소재 (**가장 시급**)

### 배경
부록 A 확정 파라미터에 `va_slope_threshold: 0.005` (0.5%) 가 존재. 이는 Regime Detection에서 Accumulation ↔ Markup/Markdown 구분에 사용됨.

그러나:
- **PLAN.md 부록 H-1 (Volume Profile 계산 명세)** 에 `va_slope` 계산 공식이 **명시되어 있는지 불확실**.
- 현재 구현체 [main.py:296](vwap_trader/src/vwap_trader/main.py#L296) 은 `va_slope=0.0` **하드코딩** 상태로 RegimeDetector 호출 → 부록 A 임계와 사실상 무관하게 작동 중.

### 질문
1. 부록 H-1 에 `va_slope` 의 계산 공식이 포함되어 있는가? (PLAN.md 2741~2808 줄 재검토 요청)
2. 만약 있다면: 현재 구현은 명백한 버그 → Dev-Core(이승준) 에게 구현 명령 필요.
3. 만약 없다면: 회의 #2 (indicators) 에서 이지원이 어느 수준까지 명시했는지 확인 + **명세 보강 회의 재개** 필요.
4. `va_slope` 는 1H 기준인가 4H 기준인가? lookback window 는 몇 봉인가?

### 영향
- 본 항목 미해결 시 **Phase 1 Regime Grid Search 결과가 의미 없음** (Regime 판정 자체가 부정확). **Phase 1 실행 전 필수 해결**.
