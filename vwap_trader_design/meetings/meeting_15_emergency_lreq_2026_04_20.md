### 1-2. 이지원 (C) 답변

```python
VA_slope = (현재 POC - 7일전 POC) / 7일전 POC
```

명세 상세:

| 항목 | 값 |
|---|---|
| 계산 기준 시간프레임 | **1H** (매 1H 봉 close 시 재계산) |
| Lookback | **7일 = 168시간** (Volume Profile 집계 창과 동일) |
| 기준점 | 현재 POC — 현재 시점 직전 168봉 VP 의 POC |
| 비교점 | 7일전 POC — 현재 시점 336봉 전부터 169봉 전까지 VP 의 POC |
| 임계 (저변동) | `abs(va_slope) < 0.005` |
| 임계 (추세) | `va_slope > +0.005` 또는 `va_slope < -0.005` |

즉 **"7일 간격 POC 변화율"** 입니다. 단순합니다.

## 2. 질의 1 — CCXT 지정의 강제성 (Agent E 주도)

### 2-1. 한지훈 (E) 답변

**한지훈**: 부록 L.1 원문은 `"데이터 소스: CCXT (bybit) + Parquet 로컬 캐싱"` 입니다.

저는 이 문구를 작성한 책임자로서 의도를 설명드리면:

- **CCXT 명시 의도**: "거래소 중립적 인터페이스로 Bybit 데이터를 수집한다" 는 **추상 의도** 였습니다. 회의 #13 (백테스트 설계) 에서 "다른 거래소로 확장 가능한 구조" 를 언급했던 이지원의 발언이 근거.
- **Parquet 명시 의도**: "바이너리 압축 포맷으로 캐싱" — 이것도 의도는 "CSV 보다 빠른 로딩", 실체가 Parquet 필수는 아님.

그러나 문서에는 **"CCXT"** 라고 고유명사가 적혀 있으므로, 문자 그대로 해석하면 강제입니다. 제 원칙상 이것을 그대로 두면 **"기록과 구현의 불일치"** 가 되어 제 역할 위반입니다.

### 2-2. 최서연 (D) 실용적 제안

**최서연**: 실용주의 관점에서 말씀드립니다. 현재 프로젝트는 **Bybit 단일 거래소 전용** 입니다. 다거래소 확장은 PLAN.md 어디에도 로드맵으로 없습니다. CCXT 를 강제할 **실익이 없습니다**.

pybit 의 장점:
- 이미 구현체가 이것으로 작성됨 ([infra/bybit_client.py](../../vwap_trader/src/vwap_trader/infra/bybit_client.py))
- Bybit v5 API 네이티브 지원 (CCXT는 wrapper 라 최신 API 반영이 느림)
- 헤지 모드 / isolated margin 전환 같은 Bybit 특유 기능 직접 지원

Parquet 도 마찬가지 — pyarrow 의존성 추가 부담 vs CSV 로도 충분한 데이터 크기. 3년 1H ~ 26K 행, CSV 로 수 MB.

### 2-3. 윤세영 (F) 판결 — 질의 1

> **판결 1-1**: 부록 L.1 의 `"CCXT (bybit)"` 표현은 **"Bybit 데이터 수집 구현체"** 로 재해석한다. pybit 사용을 공식 인정.
>
> **판결 1-2**: Parquet 은 **"로컬 바이너리/텍스트 캐시"** 로 재해석. CSV 허용.
>
> **판결 1-3**: 한지훈(E)은 부록 L.1 문구를 **"데이터 소스: Bybit API 공식 클라이언트(pybit 또는 ccxt) + 로컬 캐시(CSV/Parquet)"** 로 수정한다. 이는 기존 의도의 명확화이므로 재회의 불요.
