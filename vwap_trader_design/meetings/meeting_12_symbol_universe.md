## 1. 안건 1 — 허용 티어 범위

### 1-1. 이지원 주도 — VP 신뢰도 기준

**이지원**: Volume Profile은 **충분한 거래량이 없으면 의미가 없습니다.**

Jim Dalton의 Market Profile 원리: "시장이 공정가치를 발견하려면 충분한 참여자가 있어야 한다." 거래량이 얇은 심볼에서 POC는 통계적 노이즈에 불과합니다.

Tier별 VP 신뢰도:
```
Tier 1 (BTC/ETH): VP 신뢰도 ★★★★★
  - 일 거래량 수십억 USDT
  - POC/VAH/VAL이 강한 레벨로 작동
  - HVN/LVN 패턴 명확

Tier 2 (SOL, XRP, BNB 등 상위 20):
  - 일 거래량 수억~수십억 USDT
  - VP 신뢰도 ★★★☆☆~★★★★☆
  - POC는 유효, HVN 경계 일부 불분명

Tier 3 (기타 알트/신규):
  - 일 거래량 수천만 USDT 이하
  - VP 신뢰도 ★★☆☆☆
  - POC가 수시로 이동, 신호 신뢰 불가
```

**제안**: **Tier 1 + Tier 2만 허용. Tier 3 전면 제외.**

### 1-2. 박정우 동의

**박정우**: 동의. Brian Shannon의 VWAP 이론도 "충분한 유동성이 있을 때 VWAP은 의미 있는 지지/저항"이라고 명시합니다. Tier 3은 VWAP 자체가 의미 없습니다.

### 1-3. 김도현 이견

**김도현**: **Tier 3 일부 허용**을 제안합니다.

Module B 추세 추종의 알파는 **모멘텀 알트코인**에서 나옵니다. 신규 상장 후 급등하는 코인, 나라틱 이벤트(소각, 파트너십)가 있는 코인 — 이런 코인들은 Tier 3이지만 추세 품질이 BTC보다 훨씬 강합니다.

**이지원**: 김도현 씨, 그런 코인에서 VP의 POC와 VAH가 제대로 형성되지 않으면 Module A/B 모두 TP 레벨이 부정확합니다. 더 중요하게, 슬리피지 비용이 수익을 잠식합니다.

**김도현**: Module B는 트레일링으로 청산하므로 TP 레벨 정확도가 덜 중요합니다. 슬리피지는 포지션 크기를 줄이면 됩니다.

**최서연**: 슬리피지가 크면 Fixed Fractional 공식이 무너집니다. SL 체결 시 실제 손실이 계산값보다 커집니다. 시스템 전제 조건이 깨집니다.

**김도현**: ... 수용. Tier 3 제외. 다만 Tier 2의 상한을 충분히 넓게 정의해야 합니다.

## 2. 안건 2 — 최소 일 거래량 기준

### 2-1. 이지원 주도

**이지원**: 회의 #2 잠정값은 30M USDT/일입니다. 오늘 재검토합니다.

VP의 7일 누적 거래량으로 계산:
```
7일 VP에서 POC가 안정적으로 형성되려면:
  7일 누적 거래량 ≥ X → 일 평균 ≥ X/7

POC 안정성 기준 (경험적):
  일 거래량 20M USDT → POC 이동이 잦음, 신호 약함
  일 거래량 50M USDT → POC 안정적, HVN/LVN 명확
  일 거래량 100M+ USDT → Tier 1 수준
```

**제안**: **min_volume_usdt = 50,000,000 (50M USDT/일)**

### 2-2. 박정우 동의

**박정우**: 50M 동의. 잠정값 30M보다 올려야 합니다. 30M은 Tier 3까지 포함하는 너무 낮은 기준입니다.

### 2-3. 김도현 이견

**김도현**: **20M USDT**로 낮춰야 합니다.

2025년 Bybit 영구계약 기준, 20M USDT 이상 심볼이 약 80개입니다. 50M 기준이면 약 30~40개로 줄어듭니다. Module B 추세 추종의 기회가 제한됩니다.

**이지원**: 80개는 너무 많습니다. 오히려 거래 신호가 과다 발생하여 일일 거래 횟수 제한(RiskManager)에 먼저 걸립니다. 질 낮은 신호가 질 좋은 신호를 밀어냅니다.

**김도현**: 그건 필터 로직 개선으로 해결할 문제입니다. 유니버스를 좁히는 것은 기회 손실입니다.

**최서연**: 이지원 씨 맞습니다. 유니버스가 넓을수록 "더 좋은 신호 선택"이 필요한데, 그 로직이 현재 없습니다. 30~40개 심볼로 시작하고 백테스트 후 확장이 맞습니다.

## 3. 안건 3 — 신규 상장 제외 기간

### 3-1. 이지원 주도

**이지원**: VP 계산에 필요한 최소 기간:
- 7일 VP: 7일 데이터 필요
- ATR(14) 1H: 14개 1H 캔들 = 14시간

따라서 이론적으로 7일이면 충분합니다. 그러나 **신규 상장 코인의 비정상적 변동성**이 문제입니다.

신규 상장 패턴:
- 상장 후 1~4주: 가격 발견 단계 → 극단적 변동성
- 상장 후 4~12주: 유동성 안정화
- 상장 후 12주+: 정상 패턴 형성

**제안**: **신규 상장 후 90일 제외**.

### 3-2. 박정우 이견

**박정우**: **180일 (6개월)** 제안.

이유: VWAP의 Anchored VWAP(7일 최고/최저가 앵커)은 7일이면 충분합니다. 그러나 **Regime Detection** (4H EMA200 기울기, VA 기울기 7d)에는 더 긴 이력이 필요합니다. 4H EMA200은 200 × 4시간 = 33일의 데이터가 필요합니다. 6개월은 충분한 안전 마진입니다.

**최서연**: 6개월 지지. 퀀트 시스템에서 신규 상장 코인은 데이터 부족으로 모델 예측이 불안정합니다. 안정화된 심볼만 거래해야 합니다.

### 3-3. 김도현 반박

**김도현**: **90일**이 맞습니다. 6개월 제외는 알트 모멘텀의 황금 구간을 포기하는 것입니다.

신규 상장 코인의 최대 모멘텀은 **상장 후 30~90일** 사이에 발생합니다. 90일을 기다리면 이미 가격이 고점 대비 50~80% 하락한 상태입니다.

**이지원**: 김도현 씨 말이 맞지만 그 구간이 바로 "비정상 변동성 구간"입니다. VP가 신뢰할 수 없는 바로 그 구간입니다. Module B의 Chandelier Exit이 극단적 변동성에서 오작동할 수 있습니다.

**김도현**: ... 90일. 그게 최소한의 타협입니다.

### 3-4. 안건 3 합의 — ⚠️ 부분 합의

## 4. 안건 4 — 자동 제외 카테고리

### 4-1. 이지원 주도

**이지원**: 거래량 기준과 무관하게 **구조적으로 제외할 카테고리**가 있습니다.

**제외 카테고리 제안**:

1. **스테이블코인**: USDT, USDC, DAI, BUSD 등
   - 가격이 $1 고정 → VWAP 이탈 신호가 의미 없음

2. **레버리지 토큰**: BTCUP, BTCDOWN, ETH3L, ETH3S 등
   - 일일 리밸런싱으로 장기 보유 시 가치 훼손
   - VP 패턴이 일반 코인과 완전히 다름

3. **래핑 토큰**: WBTC, WETH 등
   - 원본 자산과 가격이 거의 동일 → 중복 포지션 위험

4. **스테이킹 파생상품**: stETH, rETH, cbETH 등
   - 언스테이킹 지연으로 가격이 원본 대비 프리미엄/디스카운트 발생
   - VP 레벨이 왜곡됨

5. **meme 코인**: DOGE, SHIB, PEPE 등
   - 소셜 미디어 드리븐, VP 무의미
   - 단, 거래량 기준 충족 시 허용 논의 가능

### 4-2. 김도현 반박 (meme 코인)

**김도현**: **meme 코인 허용**해야 합니다. DOGE와 SHIB는 Tier 2 수준 거래량을 유지합니다. Module B 추세 추종에서 meme 코인 펌프가 최고 수익원입니다.

**이지원**: DOGE 예시:
```
DOGE 2024 일 거래량: 200~500M USDT (기준 충족)
그러나 가격 드라이버: 일론 머스크 트윗, Reddit 밈
VP의 POC는 트윗 전후 완전히 리셋됨
Regime Detection으로 Accumulation/Markup 판별 불가
```

**최서연**: 김도현 씨, meme 코인에서 Module A (VWAP 이탈 반전)는 거의 작동하지 않습니다. Module B의 추세도 소셜 이벤트에 의존하므로 시스템이 예측 불가능합니다.

**김도현**: Module B는 추세 방향만 따라가면 됩니다. 왜 움직이는지는 중요하지 않습니다.

**최서연**: 그러면 정반대 논리도 성립합니다. Regime Detection이 "Markup" 신호를 냈는데 트윗 1개로 즉시 "Markdown"으로 전환됩니다. CB 2연속이 순식간에 발생합니다.

**김도현**: 수용. meme 코인 제외.

## 5. 안건 5 — 심볼 유니버스 갱신 주기

### 5-1. 최서연 주도

**최서연**: 심볼 유니버스는 정적이면 안 됩니다. 코인 거래량은 시간에 따라 변합니다.

**제안**: **주 1회 갱신 (매주 월요일 UTC 00:00)**

이유:
- 일 거래량 기준을 최근 7일 평균으로 계산
- 매일 갱신하면 심볼이 들어왔다 나왔다 하는 불안정성
- 주 1회면 충분히 최신 데이터 반영

### 5-2. 박정우, 김도현, 이지원

**박정우**: 주 1회 동의.

**김도현**: 동의. 다만 새로 유니버스에 진입한 심볼은 즉시 진입 가능? 아니면 다음 주?

**최서연**: 갱신 시점에 조건을 충족하면 즉시 진입 가능. 다음 주를 기다릴 이유 없음.

**이지원**: 동의.

## 7. 통합 심볼 필터 함수

```python
import json
from datetime import datetime, timedelta, timezone
from pathlib import Path

# ─── 제외 카테고리 패턴 ───────────────────────────
EXCLUDED_PATTERNS = [
    "USDT", "USDC", "DAI", "BUSD",      # 스테이블
    "UP", "DOWN", "3L", "3S", "BULL", "BEAR",  # 레버리지 토큰 접미사
    "W",                                  # 래핑 토큰 (WBTC 등) — 별도 검증 필요
]
EXCLUDED_SYMBOLS = {
    "WBTC", "WETH", "STETH", "RETH", "CBETH",  # 래핑/스테이킹
    "DOGE", "SHIB", "PEPE", "BONK", "WIF",     # meme
    # 확장: 운영자가 추가
}

def is_symbol_in_universe(
    symbol: str,
    volume_7d_avg_usdt: float,
    listing_date: datetime,
    blacklist_path: Path = Path("config/blacklist.json"),
) -> tuple[bool, str]:
    """
    심볼이 거래 유니버스에 속하는지 판단.
    Returns: (allowed, reason)
    """
    # ─── 긴급 블랙리스트 ─────────────────────────
    if blacklist_path.exists():
        with open(blacklist_path) as f:
            blacklist = json.load(f).get("blacklisted_symbols", [])
        if symbol in blacklist:
            return False, "blacklisted"

    # ─── 제외 카테고리 ───────────────────────────
    if symbol in EXCLUDED_SYMBOLS:
        return False, "excluded_category"
    for pat in EXCLUDED_PATTERNS:
        if symbol.endswith(pat) and symbol != pat:
            return False, f"excluded_pattern_{pat}"

    # ─── 최소 일 거래량 ──────────────────────────
    # ⚠️ Agent F 판결 반영
    MIN_VOLUME_USDT = 50_000_000  # 50M 또는 20M
    if volume_7d_avg_usdt < MIN_VOLUME_USDT:
        return False, "volume_too_low"

    # ─── 신규 상장 제외 ──────────────────────────
    # ⚠️ Agent F 판결 반영
    MIN_LISTING_DAYS = 90  # 또는 180
    now = datetime.now(timezone.utc)
    if (now - listing_date).days < MIN_LISTING_DAYS:
        return False, "listing_too_recent"

    return True, "ok"
```
