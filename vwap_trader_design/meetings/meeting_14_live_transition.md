## 1. DRY_RUN 단계 설계

### 박정우 (Module A)

DRY_RUN은 **"실전과 동일한 환경에서 가상 자금 운용"**이다. 단순 신호 기록이 아니라 주문 실행까지 시뮬레이션해야 한다. 특히 Module A는 Limit 주문을 쓰는데, 가격이 Limit에 도달했다가 체결 안 되는 케이스(취소)를 정확히 시뮬레이션해야 한다.

**DRY_RUN 필수 요소**:
- 가상 잔고 (실제 잔고와 별도)
- 실제 오더북 기반 체결 시뮬레이션 (단순 가격 터치 체결 아님)
- 모든 필터 (시간대, 심볼 유니버스, 서킷브레이커) 실제 코드 경로 통과
- 실제 수수료 + 추정 슬리피지 차감

**통과 기준 제안**:
```
1단계 DRY_RUN: 최소 50건 누적, 2주 이상 운영
  → 통과 시 2단계 진입
2단계 DRY_RUN: 추가 50건, 총 100건 누적
  → 통과 시 실전 전환 검토
```

각 단계 통과 기준은 회의 #13의 Pass/Fail 지표와 동일하게 적용.

### 김도현 (Module B)

DRY_RUN 기간에 **Module A, Module B를 동시 가동**해야 한다. 순차 검증(A 먼저 → B 나중)은 시간이 너무 걸린다. 크립토는 시장이 빠르게 변한다. 동시 가동으로 인한 혼란은 초반 시스템 버그를 빨리 발견하는 장점도 있다.

Module B 관점에서 2주 최소 기간에 동의하지 않는다. **Module B는 추세가 있어야 작동**하는데, 2주 동안 Accumulation 국면만 지속될 수 있다. 최소 **3주 이상** 또는 **Module B 트레이드 최소 20건** 중 나중에 충족되는 시점까지.

### 이지원 (Volume Profile)

DRY_RUN 기간 중 **VP 계산 정확도 모니터링**을 별도로 요청한다. 1H 캔들 기반 Volume Profile 근사치가 실제 POC/VAH/VAL과 얼마나 차이 나는지 추적해야 한다. 회의 #2에서 VP 정확도 자체 평가를 3/5로 매겼는데, 이 오차가 진입 신호에 얼마나 영향을 미치는지 DRY_RUN에서 실측한다.

**VP 정확도 임계값**: POC 오차 ≤ 0.5% → 허용 / > 0.5% → 설계 재검토

### 최서연 (Risk Management)

DRY_RUN은 **서킷브레이커 작동 여부 검증**의 유일한 기회다. 실전에서 처음 서킷브레이커가 걸리면 당황스러울 수 있다. DRY_RUN에서 의도적으로:
- 연속 손실 시나리오 만들어서 서킷브레이커 작동 확인
- 일일 손실 한도 도달 시 거래 차단 확인
- FULL_HALT → 다음 날 자동 해제 확인

이를 위해 DRY_RUN 초반 1주는 **시스템 통합 테스트 모드** 운영 제안. 의도적인 시나리오로 모든 안전장치가 작동하는지 검증.

## 2. 실전 전환 조건

### 최서연 (Risk Management)

실전 전환은 **체크리스트를 전부 통과해야 하는 게이트**다. 하나라도 실패하면 전환 불가. 이 기준은 회의에서 합의하고 나중에 바꾸지 않는 것이 원칙.

**실전 전환 필수 조건 (제안)**:

```
기술적 조건:
  □ DRY_RUN 100건 이상 누적
  □ DRY_RUN EV ≥ +0.10% (실전 목표 +0.15%의 67%)
  □ DRY_RUN 승률 ≥ 48% (목표 55%의 87%)
  □ DRY_RUN MDD ≤ 12% (실전 한계 15%의 80%)
  □ DRY_RUN Walk-Forward 기간 ≥ 3주
  □ Module A, Module B 각각 최소 15건 이상

시스템 조건:
  □ 모든 서킷브레이커 작동 확인 (의도적 테스트)
  □ 시간대 필터 24시간 작동 확인
  □ 심볼 유니버스 자동 갱신 1회 이상 확인
  □ 긴급 정지 명령 응답 시간 < 5초 확인
  □ 에러 알림 (Telegram 등) 작동 확인

운영 조건:
  □ 초기 자금 규모 사전 확정
  □ 최대 손실 허용액 사전 확정 ("이 금액 잃으면 중단")
  □ 주간 리뷰 스케줄 확정
```

### 박정우 (Module A)

DRY_RUN 통과 기준을 실전 목표 대비 낮게 설정(EV 0.10% vs 목표 0.15%)하는 것에 반대한다. DRY_RUN이 실전의 70%면 된다는 논리는 위험하다. **DRY_RUN 기준 = 실전 목표와 동일**해야 한다. 표본이 적어서 기준을 낮추는 것은 논리적 오류다.

### 김도현 (Module B)

DRY_RUN 100건 기준에 현실적 문제가 있다. Module B는 추세 추종이라 Accumulation 국면 지속 시 신호가 거의 없다. 100건 중 Module B가 5건밖에 없을 수 있다. **Module별 최소 건수**를 별도로 정해야 한다.

### 이지원 (Volume Profile)

절충안: **DRY_RUN 기준을 목표와 동일하되, 표본 부족으로 불확실성이 크다는 것을 명시**. 통계적으로 100건은 95% 신뢰구간이 ±10%p 정도라 EV 0.15%의 신뢰구간이 매우 넓다. 이를 알면서도 "목표치 도달"을 보고 진입하는 것이 맞다.

| 항목 | 박정우(A) | 김도현(B) | 최서연(D) |
|---|---|---|---|
| DRY_RUN EV 기준 | 목표와 동일 (+0.15%) | 상관없음, 모듈별 건수 중요 | 목표의 67% (+0.10%) |
| Module별 최소 건수 | 불필요 | 각 15건 이상 | 각 15건 이상 |

## 3. 자금 투입 단계화

### 최서연 (Risk Management)

**점진적 자금 투입**을 강력히 권고한다. 백테스트와 DRY_RUN이 완벽해도 실전에는 예측 불가능한 요소가 있다.

**3단계 자금 투입 제안**:

```
Stage 1 (검증 단계): 실전 첫 2주
  - 초기 자금: 전체 예정 자금의 20%
  - 목표: 실제 주문 체결 품질, 슬리피지, 수수료 현실 확인
  - 통과 조건: Stage 1 EV > 0, MDD < 10%

Stage 2 (안정화 단계): 3~8주차
  - 자금: 전체 예정 자금의 50%
  - 목표: 서킷브레이커 첫 실전 경험, 국면 전환 경험
  - 통과 조건: 6주 누적 EV > 0, MDD < 12%

Stage 3 (정상 운영): 8주차 이후
  - 자금: 전체 예정 자금의 100%
  - 조건: Stage 2 통과
```

### 박정우 (Module A)

단계화 원칙에 동의. 하지만 비율 숫자는 전적으로 사용자 재무 상황에 달린 문제다. 우리가 정할 수 없다. **원칙만 정하고 구체적 금액은 사용자 결정**으로 남겨야 한다.

### 김도현 (Module B)

동의. 비율 20/50/100%는 예시로 두되, 사용자 결정 사항으로 명시.

### 이지원 (Volume Profile)

Stage 1에서 **슬리피지 실측 데이터**를 반드시 수집해야 한다. 백테스트 가정치(Tier 1 왕복 0.10%)와 실제 차이가 크면 슬리피지 모델 재보정 필요.

### 전원 합의

```
단계화 원칙 (확정):
  1. 전체 자금을 처음부터 투입하지 않는다
  2. 최소 3단계로 나눈다 (20% → 50% → 100% 는 예시)
  3. 각 단계 통과 조건은 사전 확정, 나중에 변경 불가
  4. 구체적 금액/비율은 사용자 결정

Stage 통과 공통 조건:
  - EV > 0 (최소한 마이너스 아님)
  - MDD < 각 단계 한도 (Stage 1: 10%, Stage 2: 12%)
  - 최소 기간 충족 (Stage 1: 2주, Stage 2: 6주)
```

## 4. 모니터링 대시보드

### 이지원 (Volume Profile)

대시보드는 **지금 무슨 일이 일어나고 있는가**를 즉각 알려야 한다. 지표를 너무 많이 넣으면 오히려 판단을 방해한다.

**필수 실시간 지표**:

```
[현재 상태]
  - TradingState: ACTIVE / MODULE_A_HALT / MODULE_B_HALT / FULL_HALT
  - 현재 Regime: Accumulation / Markup / Markdown / Distribution
  - 활성 포지션 수: X/2
  - 오늘 손익: +X.XX% (잔고 대비)
  - 오늘 거래 건수: N건

[성과 누적]
  - 총 누적 EV: +X.XX% (N건 기준)
  - 누적 승률: XX%
  - 누적 PF: X.XX
  - 현재 MDD: X.XX%
  - 7일 거래 빈도: 일평균 N.X건

[모듈별 성과]
  - Module A: 건수 / 승률 / EV
  - Module B: 건수 / 승률 / EV

[경고 지표]
  - 연속 손실 카운트: Module A N회 (한도 3), Module B N회 (한도 2)
  - 오늘 손실: X.XX% (한도 5%)
  - 현재 활성 심볼 수: N개
```

### 최서연 (Risk Management)

**알림 체계**가 대시보드만큼 중요하다. 대시보드는 사용자가 보는 것이고, 알림은 사용자가 못 볼 때 작동한다.

**알림 레벨 3단계**:

```
INFO (기록만):
  - 새 포지션 진입
  - 포지션 청산
  - Regime 전환
  - 일일 결산

WARNING (알림 발송):
  - 연속 손실 N-1회 (한도 직전)
  - 오늘 손실 > 3% (한도 5%의 60%)
  - Module A/B 서킷브레이커 작동
  - 심볼 유니버스 갱신 실패

CRITICAL (즉시 알림 + 로그):
  - FULL_HALT 진입
  - 오늘 손실 > 5% (한도 도달)
  - API 연결 끊김
  - 주문 실패 3회 연속
  - 포지션 불일치 감지 (시스템 vs 거래소 실제 포지션)
```

### 박정우 (Module A)

**주간 리뷰 리포트** 자동 생성 기능 요청. 매주 월요일 UTC 00:00 (심볼 유니버스 갱신 시점과 동일)에 다음 내용을 자동 생성:

```
주간 리포트:
  - 이번 주 거래 요약 (건수, 승률, PF, EV)
  - 모듈별 성과 비교
  - Regime별 시간 점유율
  - 주요 이벤트 (서킷브레이커, HALT 등)
  - 누적 성과 (시작 이후 전체)
```

### 김도현 (Module B)

**포지션별 상세 로그** 필수. 사후 분석에서 "왜 이 진입이 손실났는가"를 파악하려면:

```
포지션 로그 필수 항목:
  - 진입 시점 Regime
  - 진입 시점 VP 레벨 (POC, VAH, VAL)
  - 진입 가격, SL, TP
  - 최종 청산 이유 (SL/TP1/TP2/Chandelier/CB/TIMEOUT/MANUAL)
  - 포지션 보유 시간
  - 실현 손익 (수수료/슬리피지 포함)
  - 진입 시점의 서킷브레이커 상태
```

### 전원 합의

모든 제안 채택. 우선순위:
1. 실시간 상태 대시보드 (이지원 안)
2. 알림 체계 3단계 (최서연 안)
3. 포지션 상세 로그 (김도현 안)
4. 주간 자동 리포트 (박정우 안)

## 5. 개입 규칙 (자동/수동 경계)

### 최서연 (Risk Management)

이것이 가장 중요한 규칙이다. **SMC-Trader에서 사용자가 개별 거래를 보고 개입한 것이 최대 실수였다.** VWAP-Trader는 이를 구조적으로 방지해야 한다.

**개입 금지 원칙**:

```
절대 개입 불가 (시스템이 자동 처리):
  - 개별 진입 신호 승인/거부
  - 개별 SL 위치 조정
  - 개별 TP 위치 조정
  - 개별 포지션 수동 청산 (MANUAL 명령 제외)
  - 파라미터 즉흥 변경

개입 가능 (사전 합의된 경우만):
  - 긴급 정지 (EMERGENCY_STOP 명령)
  - 심볼 블랙리스트 추가 (/config/blacklist.json)
  - 전략 일시 중단 (유지보수, 여행 등)
  - 특수 이벤트 등록 (FOMC, CPI 등)
```

### 박정우 (Module A)

개입 가능 케이스에 **"파라미터 재검토 회의 소집"**을 추가해야 한다. 누적 50건마다 공식 리뷰를 통해 파라미터 변경이 필요한지 검토하는 절차. 이 절차 없이 파라미터를 바꾸는 것은 금지.

```
파라미터 변경 절차 (유일한 합법적 경로):
  1) 50건 누적 리뷰에서 특정 파라미터 문제 식별
  2) 에이전트 회의 소집 (이 회의 형식과 동일)
  3) Agent F 판결
  4) 새 파라미터로 추가 DRY_RUN (최소 20건)
  5) DRY_RUN 통과 시 실전 적용
```

### 김도현 (Module B)

MANUAL 청산 조건 명확화 필요. "긴급 정지"가 개입 가능이라면 어떤 경우에만 허용되는가:

```
MANUAL 청산 허용 조건 (전원 합의):
  - 거래소 점검 공지 (포지션 위험)
  - 해킹 의심 (거래소 또는 계정)
  - 본인 또는 가족 응급 상황
```

## 6. 폐기 기준

### 최서연 (Risk Management)

폐기 기준은 **감정이 개입할 수 없게 수치로만** 정의해야 한다. "성과가 나빠 보인다" 같은 주관적 판단으로 폐기하면 안 된다. 반대로 "아직 포기하기 이르다"도 수치 없이 쓰면 안 된다.

**즉시 폐기 트리거 (Immediate Halt)**:
```
다음 중 하나라도 발생 시 즉시 운영 중단:
  1. DRY_RUN 50건 이내 MDD > 20%
  2. 실전 Stage 1 (2주) MDD > 10%
  3. 실전 Stage 2 (6주) MDD > 12%
  4. 시스템 버그로 인한 의도치 않은 손실 발생
  5. API 반복 오류로 포지션 통제 불가
```

**재검토 트리거 (Review Required — 2주 내 회의 소집)**:
```
다음 중 하나라도 해당 시 50건 리뷰 대기 없이 즉시 회의:
  1. 100건 누적 EV < 0
  2. 100건 누적 승률 < 40%
  3. 특정 모듈 50건 개별 EV < 0
  4. 국면 판별 오판 손실 > 총 손실의 40%
  5. 일 거래 빈도 3일 연속 0건 (시스템 문제 가능성)
```

**전략 폐기 기준 (Permanent Shutdown)**:
```
재검토 회의 후 다음 조건 해소 불가 시 폐기:
  - 설계 변경 없이 위 재검토 트리거 3회 이상 반복
  - 재검토 후 추가 50건에서도 EV < 0 지속
  - Module A, B 모두 개별 폐기 조건 해당
```

### 박정우 (Module A)

**Module A 단독 폐기 조건**도 필요하다. 전략 전체 폐기는 너무 극단적. 모듈 하나를 비활성화하고 나머지로 운영하는 중간 단계가 있어야 한다.

```
Module A 비활성화 조건 (100건 기준):
  - Module A 개별 EV < 0
  - Module A 승률 < 40%
  → Module A 비활성화, Module B 단독 운영 검토
```

### 김도현 (Module B)

Module B도 동일:
```
Module B 비활성화 조건 (100건 기준):
  - Module B 개별 EV < 0
  - Module B 개별 MDD > 15%
  → Module B 비활성화, Module A 단독 운영 검토
```

### 이지원 (Volume Profile)

**Regime Detection 오작동 폐기 조건** 추가 필요. VP나 모듈 자체 문제가 아니라 국면 판별이 잘못될 경우:

```

  - Distribution 국면인데 전략이 진입하는 케이스 > 5%
  - 국면 평균 유지 시간 < 4H (깜빡거림 심각)
  - 국면별 성과가 반대 방향 (Accumulation에서 Module B 성과 > Module A)
```

## 7. 긴급 정지 프로토콜

### 최서연 (Risk Management)

긴급 정지는 **"지금 당장 모든 것을 멈추는"** 명령이다. 실제 구현 필요.

```python
class EmergencyStop:
    """
    긴급 정지 프로토콜
    트리거: EMERGENCY_STOP 명령 (수동) 또는 자동 조건
    """
    
    def execute(self, reason: str):
        # 1. 새 진입 신호 차단 (즉시)
        self.block_new_entries()
        
        # 2. 기존 포지션 처리 (선택: 유지 or 즉시 청산)
        if self.is_catastrophic(reason):
            self.close_all_positions_market()   # 시장가 전량 청산
        else:
            self.maintain_existing_with_sl()    # 기존 SL 유지, 신규 금지
        
        # 3. 알림 발송
        self.send_critical_alert(reason)
        
        # 4. 상태 기록
        self.log_emergency_event(reason, timestamp=now())
    
    def is_catastrophic(self, reason: str) -> bool:
        """즉시 전량 청산이 필요한 경우"""
        catastrophic_reasons = [
            "api_connection_lost",
            "account_hacked_suspected", 
            "exchange_maintenance_emergency",
            "critical_system_bug",
        ]
        return reason in catastrophic_reasons
```

**즉시 전량 청산 vs SL 유지**: 기본값은 "SL 유지 + 신규 차단". 시장가 전량 청산은 슬리피지가 크므로 재앙적 상황에만.

### 박정우, 김도현, 이지원

긴급 정지 설계에 모두 동의. 추가 사항 없음.

### 전원 합의

```
긴급 정지 실행 시:
  Default: 신규 진입 차단 + 기존 포지션 SL 유지
  Catastrophic: 시장가 전량 청산

긴급 정지 해제 조건:
  - 수동 명령으로만 해제 (자동 해제 없음)
  - 해제 전 원인 파악 및 기록 필수
```
